# SDK releases — consuming and publishing the shell ABI

The shell is released like an SDK: one locked static implementation per
board per boundary version, published as a **tagged GitHub release with
binary assets**. This page tells a core developer what to download and a
maintainer how to cut a release. How to actually build a core against the
SDK is `RM-CONVERSION-GUIDE.md`; this page maps its "§3 inputs" onto the
release assets.

## For core developers

### What to download

From the release matching your board and boundary version (e.g. `r6-v5`):

| Asset | Why you need it |
|---|---|
| `static_locked.dcp` | **The ABI.** Every child link opens it (`make BOARD=r6 child ...`). Must be byte-identical for everyone in a catalog — check the sha256 |
| `config_a_routed.dcp` | The `pr_verify` reference your routed config is verified against |
| `config_menu.bit` | Install image: shell + menu core. Get your R6 running the shell with this |
| `config_democore.bit` | Alternative install image (shell + M2M democore), the JTAG fallback |
| `config_menu_pblock_RM_partial.bin`, `config_democore_pblock_RM_partial.bin` | Known-good partials — sanity-check your load path with these before debugging your own RM |
| `sha256sums.txt`, `RELEASE-INFO.txt` | Checksums + the release's part/boundary/Vivado identity |

You'll also want the sibling repos: this one (flow, tools, docs), the
MiSTer2MEGA65 fork branch `dfx-v5-r6` (the RM framework you vendor:
`rm_top_r6.vhd`, the `*_rm` twins, `clk_drp_master`, `qnice-rm.xdc`), and
the worked examples — `picorv32-menu` (non-M2M) and the Moon Patrol fork
(M2M, `CORE/r6-dfx-build.tcl`).

### Vivado: exact version required

Build with **exactly the Vivado version named in `RELEASE-INFO.txt`**
(currently 2023.2), for the *whole* RM flow — OOC synthesis, child link,
and bitstream generation. This is a hard DFX rule, not a preference:

- AMD requires all configurations of a DFX design — static and every
  partial — to be implemented and have bitstreams generated by the same
  Vivado release (UG909). A partial written by another version may load
  into the shell and still not run: frame-level incompatibilities fail in
  the FPGA's config engine, the least debuggable layer there is.
- A newer Vivado will *open* the locked static with only a warning and
  everything will appear to work — this is a silent failure mode.
- `pr_verify` across versions proves nothing: the reference checkpoint is
  upgraded in memory before comparison.

The cost is zero: Vivado ML **Standard** (free) covers the XC7A200T, and
Vivado versions install side by side. The ABI identity is the pair
(sha256 of `static_locked.dcp`, Vivado version); it only ever changes with
a boundary version bump, which invalidates all partials anyway.

Note there are no Vivado project files anywhere in this flow — everything
is non-project batch mode, driven by `flow/dfx.tcl` / your core's build
script.

### Hardware: two development tiers

- **With a TE0790 JTAG/serial module** on the R6's debug header
  (recommended): full dev loop — JTAG push of full bitstreams,
  `tools/send_partial.py` at 2 MBd for partials, UART status echo from the
  SD load path, the loader's diagnostics on your terminal.
- **Without one**: flash the install image once (`.cor` via the MEGA65
  flasher where provided, else ask someone with JTAG), then iterate by
  putting fresh partial `.bin`s on the SD card and loading them through
  the menu. Works, but you lose the host-visible diagnostics — budget
  more patience for debugging a core that comes up dark. The shell's LED
  verdict (BOUNDARY.md §v5) is then your main signal.

## For maintainers: cutting a release

1. Build the static and catalog for the board (`make BOARD=r6 static
   SEED_RM_DCP=...`, then the child configs); menu must be built with
   baked-in firmware (`rm_synth_r6 UART_BOOT=false`).
2. Run `pr_verify` across the whole catalog (`make BOARD=r6 verify ...`).
3. Package: `cd tools/sdk && VERSION=v5 ./package-sdk.sh` → fills
   `releases/sdk-r6-v5/` and prints the ABI hash. `RELEASE-INFO.txt` and
   `sha256sums.txt` are tracked in git (the ABI registry); the binaries
   are not.
4. `.cor` for JTAG-less installation: convert `config_menu.bit` with
   mega65-tools' `bit2core` (mega65r6 target) and add it to the assets —
   not yet automated.
5. Tag `<board>-<version>` (e.g. `r6-v5`), create the GitHub release,
   upload everything in `releases/sdk-r6-v5/`, and paste
   `RELEASE-INFO.txt` into the release notes together with the boundary
   changelog (`BOUNDARY.md`).
6. Push the matching cross-repo branches (M2M fork `dfx-v5-r6`,
   `picorv32-menu`, core forks) so the vendored framework and worked
   examples referenced by the guide are actually reachable.

Remember the SDK model's one rule: a new release invalidates **every**
existing partial. Never re-upload a changed `static_locked.dcp` under an
existing tag — that is a new boundary version by definition.
